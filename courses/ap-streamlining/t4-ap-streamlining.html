Document Change Rules — Header FieldsReglas de cambio — campos de cabecera
Configura qué campos del Document Header pueden modificarse post-contabilización, y bajo qué condiciones.
Document Change Rules = reglas de modificabilidad: SAP permite al configurador definir, por Company Code, qué campos de la cabecera de un documento FI pueden ser cambiados después de contabilizarse, y si el cambio requiere período abierto o partida no reconciliada. Los campos con impacto contable directo (Document Number, Amount, Posting Date, Company Code, Document Type) son siempre inmutables por el sistema base y no pueden habilitarse aquí.

Document Change Rules — Line Item FieldsReglas de cambio — campos de posición
Los campos de las partidas que pueden actualizarse — especialmente los relacionados con pago.
Los campos de pago de las partidas son los más frecuentemente habilitados: a nivel de partida (line item), los campos de impacto contable (Account, Posting Key, Amount, Tax ID) son siempre inmutables. Los campos de pago (Payment Terms, Baseline Date, Due Date, Payment Block) pueden habilitarse para modificación bajo condición — esto es lo que permite a Jennifer hacer el cambio masivo de payment terms en las facturas abiertas de un proveedor.

FI Global Settings
Posting Period ControlControl de períodos contables
Cómo abrir y cerrar períodos contables, que es la primera condición para permitir cambios en documentos.
El período abierto es condición previa para cambios: la primera condición que SAP verifica antes de permitir un cambio en un documento FI es si el período contable está abierto (Open). El configurador gestiona esto con la transacción OB52 — abriendo y cerrando períodos por sociedad y tipo de documento. En cierre mensual, los períodos anteriores se bloquean para prevenir modificaciones retrospectivas.

FI Global Settings
Document Types for Recurring EntriesTipos de documento para entradas recurrentes
El tipo de documento que SAP asigna a los FI documents generados por el Recurring Entry Program.
El Recurring Entry Program usa tipos documentales específicos: cuando el Recurring Entry Program genera documentos FI desde las plantillas, asigna un Document Type específico. El tipo de documento para el Original Document (la plantilla) suele ser X1 o similar — especial porque no es un asiento contable real, es solo la plantilla fuente. Los FI documents generados para cada período llevan el tipo original del documento de factura (ej. KR, RE).

Basis / System
Background Job — Recurring Entry ProgramScheduling del programa de entradas recurrentes
Configurar la ejecución automática periódica del Recurring Entry Program como background job.
El scheduling evita que nadie tenga que iniciarlo manualmente: el Recurring Entry Program puede ejecutarse de forma manual (un usuario lo inicia) o como background job programado. Un empleado autorizado configura el job: define la variante de selección (qué sociedad, qué tipo de documento), el usuario bajo cuyo nombre corre, y el horario (ej. primer día hábil de cada mes a las 06:00). Desde entonces, el programa genera los documentos automáticamente sin intervención.

Field Status Groups — Invoice Entry ScreenGrupos de field status para la pantalla de facturas
Controla qué campos son visibles, obligatorios u opcionales en la pantalla de creación de facturas.
Field Status Groups definen la pantalla de Jennifer: la experiencia de Jennifer al crear facturas en Fiori está determinada por los Field Status Groups asignados a cada GL account. El Field Status Group define si cada campo de la pantalla es obligatorio (Required), opcional (Optional) o suprimido (Suppressed). Una cuenta de gastos puede requerir Centro de Costo obligatoriamente; una cuenta de GR/IR puede suprimir ese campo.

FI Global Settings
Number Ranges for AP DocumentsRangos de numeración para documentos AP
Los rangos que controlan la numeración de los documentos FI generados por el Recurring Entry Program y las importaciones.
Los recurring docs y los docs importados necesitan rangos propios: los documentos generados por el Recurring Entry Program usan el número range asignado al Document Type del template (X1) y al tipo del documento generado (KR). Los documentos creados por Import (tipo RE o KR) usan los mismos rangos que las facturas manuales. El configurador se asegura de que los rangos tengan suficiente capacidad para el volumen de entradas recurrentes e importaciones.

Authorization — Mass Change OperationsAutorización para cambios masivos
El objeto de autorización que controla quién puede ejecutar cambios masivos en partidas de proveedores.
Mass Change requiere autorización específica: no todo usuario de AP puede ejecutar cambios masivos en partidas de proveedores. El cambio masivo puede afectar cientos de documentos simultáneamente — un error podría ser muy difícil de revertir. Por eso existe un objeto de autorización específico que controla quién puede usar la función de Mass Change en Manage Supplier Line Items, separado del permiso para cambios individuales.

FI Global Settings
Validation Rules for Document ChangesReglas de validación para cambios en documentos
Cómo configurar validaciones adicionales que SAP aplica antes de aceptar un cambio en un campo.
Validations añaden control extra más allá de Document Change Rules: además de las Document Change Rules (que definen qué campos pueden cambiarse), SAP permite configurar Validation Rules que ejecutan una lógica adicional antes de aceptar el cambio. Por ejemplo: "si el campo Payment Terms se cambia y el importe es mayor a EUR 10.000, requerir autorización adicional". Las Validations son mensajes de error o advertencia configurables por el equipo de FI.

FI — Accounts Payable
User Settings — System-Level DefaultsParámetros de usuario — nivel de sistema
La diferencia entre los parámetros configurables por el usuario (Edit Supplier Invoice Settings) y los valores por defecto definidos a nivel de sistema por el configurador.
Dos niveles de parametrización de usuario: los parámetros que Jennifer configura en Edit Supplier Invoice Settings son específicos de usuario y pueden ser modificados por ella misma. Pero el configurador puede también definir valores por defecto a nivel de sistema (SU3 / transaction defaults) que aplican a todos los usuarios de un rol, y que el usuario puede o no sobreescribir según cómo estén configurados. Esto permite estandarizar comportamientos para el equipo de AP completo.
